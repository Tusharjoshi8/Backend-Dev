//Exercise 1: Database Selection//
Choose SQL or NoSQL with justification:

1. Online Banking System → SQL
Requires strong consistency and ACID properties.
Structured data (accounts, transactions).
Example: MySQL, PostgreSQL.


2. Social Media Platform → NoSQL
Handles large-scale, unstructured data (posts, likes, comments).
Needs high scalability and flexibility.
Example: MongoDB, Cassandra.


3. Real-time Chat Application → NoSQL
High write throughput and real-time data handling.
Flexible schema for messages.
Example: MongoDB, Firebase.


4. Hospital Patient Records → SQL
Critical data integrity and consistency.
Structured relationships (patients, doctors, treatments).
Requires strict transactions.


5. IoT Sensor Data Collection → NoSQL
Massive volume of time-series data.
High scalability and fast writes.
Example: MongoDB, InfluxDB.



//Exercise 2: CAP Theorem Analysis//
(CAP = Consistency, Availability, Partition Tolerance)

1. Stock Trading Platform → CP (Consistency + Partition Tolerance)
Accuracy is critical (no incorrect trades).
Can sacrifice availability temporarily.


2. Content Delivery Network (CDN) → AP (Availability + Partition Tolerance)
Must serve content even during failures.
Slight inconsistency is acceptable.


3. Airline Booking System → CP (Consistency + Partition Tolerance)
Avoid double booking seats.
Consistency is more important than availability.


4. Video Streaming Service → AP (Availability + Partition Tolerance)
Continuous streaming is prioritized.
Minor inconsistencies (buffering, delay) are acceptable.

ASSESSMENT QUESTIONS


1. SQL vs NoSQL (Line-by-Line Differences)
Data Model:
SQL → Relational (tables, rows, columns)
NoSQL → Non-relational (documents, key-value, graph, wide-column)
Schema:
SQL → Fixed, predefined schema
NoSQL → Dynamic, flexible schema
Scalability:
SQL → Vertical scaling (increase server power)
NoSQL → Horizontal scaling (add more servers)
Data Storage Format:
SQL → Structured data
NoSQL → Semi-structured / unstructured data
Transactions:
SQL → Strong ACID compliance
NoSQL → BASE (eventual consistency in many cases)
Query Language:
SQL → Uses standard SQL (Structured Query Language)
NoSQL → Uses database-specific queries (e.g., MongoDB queries)
Performance:
SQL → Better for complex joins and transactions
NoSQL → Better for large-scale, distributed data
Examples:
SQL → MySQL, PostgreSQL, Oracle
NoSQL → MongoDB, Cassandra, Redis 


2. CAP Theorem Explanation
 CAP Theorem States:

A distributed system can only guarantee two out of three properties:

Consistency (C) → All nodes see the same data at the same time
Availability (A) → Every request gets a response
Partition Tolerance (P) → System works despite network failures
Why not all three?
In real-world distributed systems, network failures (partitions) are unavoidable.
When a partition occurs, the system must choose:
Consistency (CP) → May reject requests
Availability (AP) → May return stale data
Therefore, achieving all three simultaneously is impossible.


3. When MongoDB is Preferred Over Relational Databases ?
Social Media Applications
Handles unstructured data like posts, comments, reactions
Flexible schema for evolving features
Real-Time Analytics / Big Data
High-speed data ingestion (logs, metrics)
Scales horizontally across servers
Content Management Systems (CMS)
Stores varied content formats (text, images, metadata)
No need for rigid schema changes

4. Why MongoDB Uses BSON Instead of JSON ?
MongoDB uses BSON (Binary JSON) internally because:
Faster Processing → Binary format is quicker to parse
Rich Data Types → Supports Date, ObjectId, Decimal128, etc.
Efficient Storage → More compact than plain JSON
Better Query Performance → Optimized for indexing and traversal
JSON is human-readable, but BSON is machine-efficient.
